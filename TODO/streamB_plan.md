# Stream B Plan: Abstract Program Synthesis via LLM

**Objective:** Given C code + data-annotated assertions + abstract program signatures (from Stream A pipeline), use an LLM to synthesize the Coq monadic bodies for `{func}_M`, `{func}_M_loop`, and `{func}_M_loop_end`, then verify them with `coqc`.

---

## Phase 1: LLM-Based Abstract Program Synthesis

### Background

The Stream A pipeline already generates:
- Translated `_rel.c` files with `safeExec`/`bind` wrappers in annotations
- Abstract program **signatures** as `Extern Coq` declarations (e.g., `sll_copy_M: list Z -> program unit (list Z)`)
- QCP's symexec generates VCs (`_goal.v`) referencing these programs by name

What's **missing** is the `_rel_lib.v` file containing the actual Coq **Definition** bodies for these programs. Currently these are axiomatized — they need concrete implementations.

### 1.0 Understand the Target Format

#### Abstract Program Decomposition (from paper §3.3)

Each function's abstract program decomposes into **5 sub-components**:

| Component | Type | Role |
|-----------|------|------|
| `f_M_loop_before` | `ArgType1 → ... → ArgTypen → M(S)` | Maps Require variables to initial invariant state `S` |
| `guard` | `S → Prop` | Loop guard (already generated by GuardGen) |
| `M_1` (break) | `S → M(R)` | Executed when guard is **false** (loop exits) |
| `M_2` (continue) | `S → M(S)` | Executed when guard is **true** (one iteration step) |
| `f_M_loop_end` | `R → M(RetType1 × ... × RetTypem)` | Maps loop result `R` to Ensure variables |

Where `S = LoopArgType1 × ... × LoopArgTypek` (k-tuple of invariant type vars), `R = MretTy` (opaque return type), and `M = program unit`.

#### Loop Structure: `repeat_break`

The loop model `f_M_loop` is constructed via QCP's `repeat_break`, which repeatedly executes a body that either continues or breaks:

```
guard   : S → Prop
M_1     : S → M(R)           // break branch
M_2     : S → M(S)           // continue branch

function f_M_loop(a) =
  repeat {
    choose
      assume ¬guard(a);; r ← M_1(a);; break r
    or
      assume guard(a);; a' ← M_2(a);; continue a'
    end
  }
```

#### Composition

The full abstract program composes these via `bind`:

```
f_M(l1, ..., lm) =
  s₀ ← f_M_loop_before(l1, ..., lm);;
  r  ← f_M_loop(s₀);;
  f_M_loop_end(r)
```

In the C annotations this appears as:
- **Require**: `safeExec(ATrue, f_M(l1, ..., lm), X)`
- **Inv**: `safeExec(ATrue, bind(f_M_loop(l1, ..., lk), f_M_loop_end), X)`
- **Ensure**: `safeExec(ATrue, return(l_{m+1}, ..., l_n), X)`

#### Synthesis Target

The synthesis target is a `{basename}_rel_lib.v` file. The **LLM must generate the 4 non-guard components** (`f_M_loop_before`, `M_1`, `M_2`, `f_M_loop_end`), since the guard is already produced by GuardGen. These are composed into `f_M_loop` and `f_M` using the fixed `repeat_break`/`bind` scaffolding above.

**Concrete example — `sll_reverse`** (from paper, commented-out section):

```
// f_M_loop_before: ?l1 ↦ ([], ?l1)
//   prev starts empty, curr starts with full list

// M_2 (continue): (l1, l2) ↦ (h :: l1, t)  where l2 = h :: t
//   one reversal step: move head of curr to front of prev

// M_1 (break): (l1, l2) ↦ l1
//   loop exits when l2 = [], result is the reversed list l1

// f_M_loop_end: r ↦ (r)
//   identity — loop result is the ensure variable directly
```

Key observations:
- `MretTy` is an opaque type — its concrete definition (a product of lists) goes in the `_rel_lib.v`
- `M_2` is the core semantic piece — it models what one C loop iteration does to the abstract list state
- `M_1` is typically simple — just packages the final state when the guard fails
- `f_M_loop_before` maps function inputs to the initial invariant state (often straightforward from comparing Require vars to invariant vars)
- `f_M_loop_end` maps the opaque loop result to the Ensure vars (often identity or simple projection)
- The list operations in `M_2` must be consistent with what the C loop body does to the heap

### 1.1 Define Input/Output Format

**Input to the LLM (structured prompt):**

| Component | Source | Example |
|-----------|--------|---------|
| C source function | `shape_invdataset/` | The full function body |
| Data-annotated Require/Ensure | `_rel.c` output | `sll(x, l1)` → `sll(__return, l2) * sll(x, l3)` |
| Data-annotated loop invariant | `_rel.c` output | `sllseg(x@pre, p, l1) * sll(p, l2) * sllseg(y, t, l3)` |
| Loop condition | C source | `p` (i.e., `p != NULL`) |
| Guard function (Coq) | GuardGen output | `fun l2 => match l2 with nil => false \| _ => true end` |
| Abstract program signatures | `Extern Coq` block | `sll_copy_M: list Z -> program unit (list Z)` etc. |
| Functional spec (if provided) | User-supplied or inferred | `result = input` (copy), `result = rev(input)` (reverse) |
| Few-shot examples | Curated library | 2-3 solved examples of similar pattern |

**Output from the LLM:**

The 4 non-guard sub-components (guard is already generated by GuardGen):
1. `f_M_loop_before` body: how Require vars map to initial invariant state `S`
2. `M_2` body (continue branch): how one loop iteration transforms `S → S`
3. `M_1` body (break branch): how final state maps to loop result `S → R`
4. `f_M_loop_end` body: how loop result maps to Ensure vars `R → output`

These are assembled into `MretTy`, `{func}_M_loop`, `{func}_M_loop_end`, and `{func}_M` definitions in a `{basename}_rel_lib.v` file using the fixed `repeat_break`/`bind` scaffolding.

### 1.2 Build Prompt Templates

Create a prompt template system in `GenMonads/absprog/` (new module):

#### Task 1.2.1: Core prompt template
Design a structured prompt that presents:

```
You are generating Coq monadic abstract programs for formal verification.

## C Function
{c_source}

## Data Annotations (translated)
Require: {require_annotation}
Ensure: {ensure_annotation}
Loop Invariant: {loop_invariant}
Loop Condition: {loop_condition}

## Abstract Program Signatures
{extern_coq_block}

## Guard Function
{guard_coq}

## Functional Specification
{func_spec}  (* e.g., "The output list equals the input list" *)

## QCP Monad Primitives
- `return v` / `ret v`: return value v (monadic return)
- `bind m f` / `m >>= f`: sequence m then f
- `program unit T`: monadic program returning T (StateRelMonad)
- `assume P` : enforce the condition P of type Prop holds and returns the unit value tt
- `any A` : return an value v of type A
- `choice m1 m2` : nondeterministic branching
- `repeat_break`: loop construct with break and continue branch
- List operations: `app` (++), `cons` (::), `nil`, `length`, etc.

## Abstract Program Decomposition
The loop model uses `repeat_break` with two branches:
- **M_2 (continue)**: when guard is true, one iteration step `S → M(S)`
- **M_1 (break)**: when guard is false, produce final result `S → M(R)`

The full program composes:
  f_M_loop_before → f_M_loop (repeat_break with M_1, M_2, guard) → f_M_loop_end

## Instructions
Generate the 4 non-guard components:
1. **f_M_loop_before**: How do the Require variables map to the initial invariant
   state? (e.g., for sll_reverse: `?l1 ↦ ([], ?l1)` — prev starts empty, curr gets full list)
2. **M_2 (continue branch)**: What does one C loop iteration do to the abstract list
   state? (e.g., for sll_reverse: `(l1, h::t) ↦ (h::l1, t)` — move head to accumulator)
3. **M_1 (break branch)**: When the loop exits, how is the final result packaged?
   (e.g., for sll_reverse: `(l1, []) ↦ l1` — return the accumulated list)
4. **f_M_loop_end**: How does the loop result map to the Ensure variables?
   (e.g., for sll_reverse: `r ↦ (r)` — identity mapping)

## Examples
{few_shot_examples}
```

|---------|---------|---------------------|-------------|

#### Task 1.2.3: Pattern classification heuristic
Before calling the LLM, classify the function into a pattern category to select the most relevant few-shot examples:

- Count of list variables in invariant → arity of MretTy
- Presence of `lseg`/`sllseg` → traversal or accumulation pattern
- Number of `listrep`/`sll` in Require vs Ensure → copy/split vs in-place
- Number of loops → single vs multi-loop

### 1.3 Extract Synthesis Inputs from Pipeline

Extend the existing pipeline to collect all information needed for the LLM prompt.

#### Task 1.3.1: `collect_synthesis_context()` function
Implemented in `GenMonads/absprog/context.py`.

Current API:
- `collect_synthesis_context(c_file, func_name=None) -> Dict`
- `collect_all_synthesis_contexts(c_file) -> List[Dict]`
- `write_synthesis_context(c_file, output_path, func_name=None) -> Dict`

Behavior:
- For single-function files, `collect_synthesis_context()` can be called with just the C file path.
- For multi-function files, `collect_synthesis_context()` requires `func_name`.
- `collect_all_synthesis_contexts()` returns one context per synthesizable function and skips helper functions with no loop invariants.
- Directory batch generation is exposed via `uv run llm4pv-context <input-file-or-dir> <output-file-or-dir>`.

### 1.4 LLM Integration

#### Task 1.4.1: LLM caller module (`GenMonads/absprog/synthesize.py`)
- Accept the synthesis context dict
- Render the prompt template with context + selected few-shot examples
- Call Claude API (or configurable LLM endpoint)
- Parse the response to extract Coq definitions
- Return structured output: `{M_loop_before, M_1, M_2, M_loop_end}` (the 4 non-guard components)

#### Task 1.4.2: Response parser (`GenMonads/absprog/parse_coq.py`)
- Extract Coq `Definition` blocks from LLM response (may be in markdown code fences)
- Validate basic structure: correct function names, correct argument counts
- Validate types match signatures from `Extern Coq` block

#### Task 1.4.3: `_rel_lib.v` file assembler (`GenMonads/absprog/assemble.py`)
- Generate complete `.v` file with:
  - Standard Coq imports (ZArith, List, monad library, sll_lib, etc.)
  - `MretTy` definition
  - All abstract program definitions
  - `Local Open Scope` directives
- Template for the boilerplate (imports are mostly fixed per predicate family)

### 1.5 Validation Gate

#### Task 1.5.1: Syntax check with `coqc`
- Run `coqc` on the generated `_rel_lib.v` to catch type errors, undefined references, syntax errors
- This is a fast, cheap check before attempting full VC proofs

#### Task 1.5.2: Structural validation (pre-coqc)
Before even running `coqc`, validate:
- `MretTy` is a product of the correct number of `list Z` components (matches invariant var count)
- `M_2` (continue) maps `S → S` (same tuple arity in and out)
- `M_1` (break) maps `S → R` (produces `MretTy`)
- `f_M_loop_before` takes the correct number of Require vars and produces `S`
- `f_M_loop_end` takes `MretTy` and returns the correct Ensure type
- All assembled definitions use the correct `repeat_break`/`choose`/`assume`/`break`/`continue` scaffolding

#### Task 1.5.3: Retry on failure
- If `coqc` fails on `_rel_lib.v`, feed the error back to the LLM with context
- Cap at 3 retries for the synthesis step alone (Phase 3 handles full VITL loop)

### 1.6 Multi-Function and Multi-Loop Support

#### Task 1.6.1: Per-function synthesis
For files with multiple functions (e.g., `sll_rotate.c` with `sll_split` and `sll_rotate`):
- Run synthesis independently per function
- Each function gets its own `{func}_M`, `{func}_M_loop`, `{func}_M_loop_end`
- All definitions go into the same `_rel_lib.v`

#### Task 1.6.2: Multi-loop handling
For functions with multiple loops (e.g., `sll_copy_rev.c`):
- Variables are already prefixed by Stream A (`l1_1`, `l1_2`, etc.)
- Need separate `M_loop_1`, `M_loop_2` etc., or a single `M` that sequences both loops
- The LLM prompt must clearly indicate loop ordering and variable scoping
