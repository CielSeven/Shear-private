# Function-Call Residual Abstract Program Plan

## Summary

We need a plan for generating explicit Rocq residual-program definitions for function-call sites, based on the current abstract-program scaffold in an existing `rel_lib` file.

The intended workflow is:

Input:

1. a generated `{file}_rel_lib.v` file
   It already gives the complete scaffold definition for the caller abstract program, including the exact types of:
   - `{func}_M_loop_before`
   - `{func}_M_loop_M1`
   - `{func}_M_loop_M2`
   - `{func}_M_loop_end`
   - `{func}_M_after_loop`, when present
2. a C file, or an equivalent structured analysis artifact such as JSON
   It tells us:
   - where the same-file opaque callee calls occur
   - which concrete call is at each position,
   - and what abstract program corresponds to the callee

Output:

1. explicit Rocq code segments such as:

```coq
Definition residual_prog_after_{func}_position_x : ... :=
  fun m ... => ...
```

These residual definitions should take the following as the arguments:

- the callee return value,
- any extra logical variables they close over

These residual definitions should

- be type-correct with respect to the actual component type in the existing `rel_lib`
- describe the remaining abstract computation after that call site


The main goal is:

- given an existing `rel_lib` scaffold and call-site information
- mechanically derive Rocq residual-program definitions for function calls

Any JSON representation should be treated only as an optional intermediate artifact for transport, debugging, or prompting.

The current scaffold can take these forms, especially in the updated loop-body-early-return cases:

- no early return:
  - `{func}_M_loop_before : ReqArgs -> MONAD S`
  - `{func}_M_loop_M1 : S -> MONAD R`
  - `{func}_M_loop_M2 : S -> MONAD S`
  - `{func}_M_loop_end : R -> MONAD Ret`
- pre-loop early return only:
  - `{func}_M_loop_before : ReqArgs -> MONAD (early_result S Ret)`
- loop-body early return only:
  - `{func}_M_loop_before : ReqArgs -> MONAD S`
  - `{func}_M_loop_M1 : S -> MONAD R`
  - `{func}_M_loop_M2 : S -> MONAD (early_result S Ret)`
  - `{func}_M_after_loop : early_result R Ret -> MONAD Ret`
- both pre-loop and loop-body early return:
  - `{func}_M_loop_before : ReqArgs -> MONAD (early_result S Ret)`
  - `{func}_M_loop_M2 : S -> MONAD (early_result S Ret)`
  - `{func}_M_after_loop : early_result R Ret -> MONAD Ret`

The purpose of this plan is:

- when a caller function invokes an opaque same-file callee
- generate a residual Rocq definition for the code that comes after that call
- derive that residual mechanically from the existing scaffold and the C control flow
- later, optionally expose that residual to the LLM

`sll_multi_merge` can be taken as an testing file, where different `sll_merge` call sites live in different semantic regions:

- pre-loop early-return branch
- loop-body continue branch
- loop-body early-return branch

Each such call site has a different residual shape.

---

## 1. Main Design Principle

Do not ask the LLM to rediscover function-call residual continuations from raw C syntax.

First derive the residual Rocq definitions mechanically.
Then, if useful, expose those derived definitions to the LLM.

Instead:

1. detect the call site mechanically
2. determine which existing scaffold component owns that call site
3. reconstruct the remaining computation from that call point
4. write it as an explicit Rocq residual definition
5. simplify the residual when the scaffold reduces immediately
6. recover the logical variables that the residual closes over
8. validate that later synthesized Coq follows that residual structure

The key point is:

> residuals must be derived from the actual generated abstract-program scaffold

---

## 2. Scope

This plan targets the current supported synthesis setting:

- one target function per prompt
- at most one loop per target function
- same-file callees are opaque and already have abstract counterparts such as `{callee}_M`
- we only consider same-file opaque calls
- the caller may have:
  - calls before the loop
  - calls inside loop-body branches
  - calls after the loop
  - pre-loop early returns
  - loop-body early returns

We only target the current one-loop function shape already supported by `llm4pv-synth`.
This plan is not trying to solve arbitrary whole-program function-call analysis.

## 3. Updated Ground Truth: Current Scaffold Shapes

Residual detection must start from the actual scaffold currently generated by rel-lib generation and shown in the synthesis context.

Let:

- `ReqArgs` be the Require-side argument types
- `S` be the loop carrier type from the invariant
- `R` be `MretTy`, the normal loop-exit result type
- `Ret` be the final function return type

### Case A: No early returns

```coq
Definition {func}_M : ReqArgs -> MONAD Ret :=
  fun ... =>
    s0 <- {func}_M_loop_before ...;;
    r <- {func}_M_loop_aux s0;;
    {func}_M_loop_end r.
```

with:

```coq
Definition {func}_M_loop_body : S -> MONAD (CntOrBrk S R) := ...
Definition {func}_M_loop : InvArgs -> MONAD R := ...
```

### Case B: Pre-loop early return only

```coq
Definition {func}_M : ReqArgs -> MONAD Ret :=
  fun ... =>
    e <- {func}_M_loop_before ...;;
    match e with
    | Continue s =>
        r <- {func}_M_loop_aux s;;
        {func}_M_loop_end r
    | ReturnNow r =>
        return r
    end.
```

### Case C: Loop-body early return only

```coq
Definition {func}_M_after_loop : early_result R Ret -> MONAD Ret :=
  fun re =>
    match re with
    | Continue r => {func}_M_loop_end r
    | ReturnNow r => return r
    end.

Definition {func}_M_loop_body : S -> MONAD (CntOrBrk S (early_result R Ret)) := ...
Definition {func}_M_loop : InvArgs -> MONAD (early_result R Ret) := ...

Definition {func}_M : ReqArgs -> MONAD Ret :=
  fun ... =>
    s0 <- {func}_M_loop_before ...;;
    re <- {func}_M_loop_aux s0;;
    {func}_M_after_loop re.
```

and the important updated part is:

```coq
Parameter {func}_M_loop_M2 : S -> MONAD (early_result S Ret).
```

so a continue-branch helper call inside the loop may now lead either to:

- `Continue s'`
- `ReturnNow r`

### Case D: Both pre-loop and loop-body early returns

```coq
Definition {func}_M_after_loop : early_result R Ret -> MONAD Ret :=
  fun re =>
    match re with
    | Continue r => {func}_M_loop_end r
    | ReturnNow r => return r
    end.

Definition {func}_M : ReqArgs -> MONAD Ret :=
  fun ... =>
    e <- {func}_M_loop_before ...;;
    match e with
    | Continue s =>
        re <- {func}_M_loop_aux s;;
        {func}_M_after_loop re
    | ReturnNow r =>
        return r
    end.
```

## 4. What "Residual Program" Means Now

For a function call site, the residual program should mean:

> the remaining scaffolded abstract computation from the point immediately after that call

and our concrete deliverable should be a named Rocq definition for that residual.

This is not always identical to "the rest of the local component only".

We should mechanically derive the remaining scaffold from the call point, and then simplify it when the scaffold wrapper immediately reduces.

### Example 1: First call in `sll_multi_merge`

C branch:

```c
if (x == 0) {
    t = sll_merge(y, z);
    return t;
}
```

Suppose the abstract call is:

```coq
m <- sll_merge_M l2 l3;;
```

Then the remaining scaffold from that call point is:

```coq
e <- return (ReturnNow m);;
match e with
| Continue s =>
    re <- sll_multi_merge_M_loop_aux s;;
    sll_multi_merge_M_after_loop re
| ReturnNow r =>
    return r
end
```

and this simplifies to:

```coq
return (ReturnNow m)
```

So the generated residual definition for this call site can simply be:

```coq
Definition residual_prog_after_sll_multi_merge_position_1
  : list Z -> MONAD (early_result S Ret) :=
  fun m => return (ReturnNow m).
```

The derivation may internally use the remaining scaffold view, but the output we want is the explicit Rocq residual definition.
